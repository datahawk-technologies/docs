Use this page when you need to query DataHawk's sales estimate tables or migrate from older V5/V6 table names. For methodology and interpretation, see [Amazon Sales Estimates](/help-center/data-metrics-guides/competition/amz-sales-estimates).

## Active tables

| Schema | Table | Level | Description |
|---|---|---|---|
| `PRODUCT` | `PRODUCT_SALES_ESTIMATES` | Child + Parent | One row per tracked child ASIN per day. Contains both variant-level and parent-rollup fields. |
| `MARKET` | `PRODUCT_SALES_ESTIMATES` | Product | Top 100 products per category with V7 estimates. |
| `REPORTS` | `REPORT_MARKET_BEST_SELLER_RANK_AND_ESTIMATES` | Product | Top 100 per category with ranks and estimates. |

| `TAGS_ARRAY` | Child | Tags applied to this product in DataHawk |

| `SYNC_TIME` |; | Last data sync timestamp |

The V7 table no longer exposes `OBSERVED_CHANNEL_PRODUCT_ID`, `RATING`, or `RATING_COUNT`.

## Granularity and refresh

| Attribute | Value |
|---|---|
| Granularity | One row per tracked child ASIN / marketplace / day |
| Refresh | Daily |
| History | Up to two full years, depending on marketplace |
| Primary key | `WORKSPACE_ID`, `OBSERVATION_DATE`, `MARKETPLACE_KEY`, `ASIN` |

The full 2024+ history was reprocessed with V7 on June 15, 2026.

## FAQ

<Accordions>
  <Accordion title="Do V5 or V6 queries still work?">
    Yes. V5 and V6 tables now resolve to V7 data via aliases, so existing queries continue to return values. The values may differ from what you saw before June 15, 2026 because V7 reprocessed the historical data.
  </Accordion>

  <Accordion title="Should I migrate queries to the new table names?">
    Yes. Aliases work today but should not be used for new work. Replace `PRODUCT_SALES_ESTIMATES_V5*` and `_V6*` references with `PRODUCT_SALES_ESTIMATES`, and use the active `MARKET` and `REPORTS` tables listed above.
  </Accordion>

  <Accordion title="Why are child estimates different from the parent total?">
    Parent estimates represent total family sales. `UNITS_PARENT` is the rolled-up value across known variants in that family on that date. When you compare a complete family for the same date, the sum of `UNITS_CHILD` should reconcile with `UNITS_PARENT`.
  </Accordion>

  <Accordion title="My child estimates don't add up to the parent. Why?">
    This can happen when the family definition changes over time, or when only some variants in a family are tracked. When you track one child ASIN from a larger family, DataHawk uses the parent ASIN to discover the family structure, but it does not collect sales signals for the untracked children.

    If the tracked child has no public purchase signal, its estimate uses the default low-signal range of 0-50 monthly units, represented as 25 monthly units. That child-level estimate can stay low even when the full parent family sells more than 30,000 units per month. Use `VARIATION_COUNT` to understand the full known family size used at estimation time.
  </Accordion>

  <Accordion title="Can I get estimates for products I don't track directly?">
    Yes. If a product appears in the top 100 of a category you track, category tracking can provide estimates for it even if you did not add that ASIN manually.
  </Accordion>

  <Accordion title="What is IS_EXTRAPOLATED?">
    When BSR data is temporarily unavailable, DataHawk fills the gap using linear extrapolation. These rows are flagged so you can include or exclude them in analysis.
  </Accordion>

  <Accordion title="What is TRUST_INDEX?">
    A per-row confidence score reflecting data quality for that product and date.
  </Accordion>
</Accordions>
