This page is the **conceptual map** of DataHawk's data. It doesn't try to list every column — that lives in the [Exhaustive Column Referential](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716) Google Sheet, which is always current and supports filter, sort, and search far better than any markdown table could. Use this page to **understand what's where and why**, then jump to the Sheet for column-level lookup.

<Callout type="info">
  The Sheet is the single source of truth for column names, data types, and per-column descriptions. This page covers the layer above — what each schema is for, how tables relate, and where to start for common questions.
</Callout>

## The Exhaustive Column Referential

The Sheet documents every table and every column across all DataHawk schemas. Each row tells you:

- **Schema** — top-level grouping (e.g. `PRODUCT`, `FINANCE`, `ADVERTISING`)
- **Table** — the table name within the schema
- **Column** — the field name as it appears in your <Term>Snowflake</Term> or <Term>BigQuery</Term> database
- **Description** — what the field represents
- **Data type** and **example value** where useful

**Working with the Sheet:**

- Use the Sheet's built-in filter to narrow by schema, table, or column name
- Use `Ctrl/Cmd + F` to search across descriptions for keywords like "tax", "deferred", "buy box"
- The Sheet has multiple tabs — switch between them via the tab bar at the bottom

**[Open the Exhaustive Column Referential →](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716)**

## Schemas at a glance

DataHawk organizes data into top-level schemas. Each schema groups related tables by their data source or purpose.

| Schema | What's in it | When to look here |
|---|---|---|
| **`ADVERTISING`** | Ad performance data across Amazon and Walmart | Campaign, keyword, product, and account-level ad metrics |
| **`FINANCE`** | Orders, profit, P&L, fees, taxes — Seller accounts only | Revenue, margin, cost-of-goods, financial events |
| **`PRODUCT`** | Product catalog, attributes, listing quality, sales estimates | Anything about a tracked product or its competitive landscape |
| **`MARKET`** | Top-100-per-category data with sales estimates | Category-level competitive analysis, market sizing |
| **`SELLING_PARTNER`** | Direct Seller/Vendor Central data — FBA inventory, vendor sales, BA reports | Detailed Amazon-account data the higher-level schemas summarize |
| **`SEO`** | Keyword rankings (organic + sponsored), search volume | Organic visibility, sponsored placement, keyword research |
| **`REPORTS`** | Pre-built consolidated reports joining data across schemas | Dashboard-style queries without writing joins yourself |
| **`REFERENTIAL`** | Reference data — tracked products, keywords, tags, accounts, currency rates | Look up "what am I tracking?", "what tags exist?", "what's the current exchange rate?" |
| **`RAW_INVENTORY`** | Raw FBA inventory snapshots and ledger events | Low-level inventory state, before DataHawk aggregation |
| **`USAGE`** *(Snowflake only)* | Credit consumption, query history | Operate / monitor your DataHawk Snowflake usage |

For the full table-and-column list inside each schema, see the [Sheet](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716).

## Conceptual map

The schemas above are an inventory. What follows are the **flows** — how data moves through them, so you know which table feeds which.

### Profit and finance: from raw events to net P&L

```
FINANCE_EVENTS                      ← raw financial events from Amazon (kept for completeness)
        ↓
FINANCE_PROFIT_LEDGER               ← deduplicated, categorized — canonical source for profit work
        ↓
FINANCE_PROFIT  /  FINANCE_PROFIT_LOSS  ← daily aggregations
        ↓
FINANCE_PRODUCT_PROFIT_LOSS         ← per-product P&L with estimated ad costs
```

When reconciling against Seller Central, **use `FINANCE_PROFIT_LEDGER`** with `transaction_status = 'RELEASED'`. The ledger also exposes `is_tax` for tax extraction, `deferral_reason` for B2B / Delivery-Date+7 distinctions, and `settlement_id` for joining to Amazon settlement reports. See the [Sheet](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716) for the full column list.

### Inventory: from inbound to fulfillable

```
RAW_INVENTORY_LEDGER_DETAIL      ← raw movement events (deprecated — use SELLER_INVENTORY_LEDGER_*)
        ↓
SELLER_INVENTORY_LEDGER_DETAIL   ← country-level granular movement events (current)
        ↓
SELLER_INVENTORY_LEDGER_SUMMARY  ← daily aggregated health
        ↓
SELLER_INVENTORY_FBA_HISTORY_DAILY  ← daily snapshot of all FBA inventory states
        ↓
SELLER_INVENTORY_FBA_PLANNING    ← AI-driven recommendations for aged stock and removals
```

The FBA inventory state machine has many states (`available`, `inbound`, `working`, `reserved`, `unfulfillable`, etc.). See the **Inventory** section of the [Power BI Seller Analytics Dashboard guide](/help-center/powerbi-dashboards/seller-analytics-dashboard) for the state definitions, or the [Sheet](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716) for per-column detail.

### Advertising: account → campaign → keyword/product

```
ADVERTISING_ACCOUNT_METRICS      ← account totals
        ↓
ADVERTISING_CAMPAIGN_METRICS     ← per-campaign
        ↓
ADVERTISING_KEYWORD_METRICS_V2   ← per-keyword × campaign × ad-group (v2 surface)
ADVERTISING_PRODUCT_METRICS      ← per-product (SP + SD only — Sponsored Brands not available at product level)
```

For Walmart specifically, see [Walmart Advertising Data](/help-center/knowledge-hub/wmt-advertising-data) — the page covers `WMT_AD_ITEM` (the current canonical, replacing the deprecated `WMT_SP_AD_ITEM`), the new `WMT_SP_ITEM_KEYWORD_V2`, and the placement/page-type tables.

### Sales estimates: variant → parent → brand → market

DataHawk's V7 estimate model (June 2026) generates daily sales estimates **natively at the variant (child <Term>ASIN</Term>) level**, with rollups to parent and brand:

```
PRODUCT.PRODUCT_SALES_ESTIMATES        ← per-child, with parent rollup columns
        ↓
MARKET.PRODUCT_SALES_ESTIMATES         ← top-100-per-category estimates
        ↓
REPORTS.REPORT_MARKET_BEST_SELLER_RANK_AND_ESTIMATES  ← consolidated view with brand metadata
```

All `_V5_*` and `_V6_*` table names are deprecated aliases — they still resolve to V7 data, but switch new queries to the canonical names. See [Amazon Sales Estimates](/help-center/knowledge-hub/amz-sales-estimates) for the full migration table.

### SEO: organic vs sponsored rank

```
SEO_PRODUCT_ORGANIC_RANK         ← daily organic ranks (top 5 pages)
SEO_PRODUCT_SPONSORED_RANK       ← daily sponsored ranks — requires Premium Keywords
SEO_KEYWORD_VOLUME_V2            ← real Amazon search volume (Search Terms Performance, Oct 2025+)
```

<Callout type="warn">
  Standard sponsored ranks were **retired on July 8, 2025**. The `SEO_PRODUCT_SPONSORED_RANK` table is now populated only for workspaces that have switched on [Premium Keywords](/help-center/data-setup/premium-keywords-tracking).
</Callout>

## How to find a column

A reliable three-step process when you know the concept but not the column name:

<Steps>

<Step>
### Identify the schema

Use the "Schemas at a glance" table above. If your question is about ads → `ADVERTISING`. About profit or P&L → `FINANCE`. About a competitor's category position → `MARKET`. About what's tracked → `REFERENTIAL`.
</Step>

<Step>
### Open the Sheet and filter

Open the [Exhaustive Column Referential](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716). Filter the Schema column to the schema you identified.
</Step>

<Step>
### Search within the filtered set

Use `Ctrl/Cmd + F` to search the descriptions for the concept you're after — for example "buy box", "deferred", "trust", "BSR". The Sheet's full descriptions are usually richer than any table-of-tables would be.
</Step>

</Steps>

If you still can't find it, ask in `#data-questions` or open the relevant Knowledge Hub page (linked under "Where to go next" below) — those pages cover concepts like Buy Box, BSR, and Sales Estimates in depth.

## Common query patterns

A few starter queries for the most-asked questions.

### Which brands dominate a tracked category?

```sql
SELECT brand, brand_catalog, COUNT(*) AS appearances
FROM reports.report_market_best_seller_rank_and_estimates
WHERE category_id = 'YOUR_CATEGORY_ID'
  AND observation_date > CURRENT_DATE - 30
GROUP BY brand, brand_catalog
ORDER BY appearances DESC;
```

### Which keywords drive your organic traffic?

```sql
SELECT keyword, AVG(rank) AS avg_rank, COUNT(*) AS days_ranked
FROM reports.report_seo_product_organic_rank
WHERE asin = 'B0XXXXXXXX'
  AND observation_date > CURRENT_DATE - 30
GROUP BY keyword
HAVING AVG(rank) < 50
ORDER BY avg_rank ASC;
```

### What's your true daily P&L?

```sql
SELECT
  observation_date,
  SUM(CASE WHEN event_category = 'Revenue' THEN amount ELSE 0 END) AS gross_revenue,
  SUM(CASE WHEN event_category = 'Refund' THEN amount ELSE 0 END) AS refunds,
  SUM(CASE WHEN is_tax THEN amount ELSE 0 END) AS taxes,
  SUM(CASE WHEN event_category = 'Fees' THEN amount ELSE 0 END) AS amazon_fees
FROM finance.finance_profit_ledger
WHERE transaction_status = 'RELEASED'
  AND observation_date > CURRENT_DATE - 30
GROUP BY observation_date
ORDER BY observation_date DESC;
```

Replace column and table names with the exact ones from the [Sheet](https://docs.google.com/spreadsheets/d/1gYGqKQP9Z71nz1lOb4y0wJ15i27NvsUzdliIJeIx0HY/edit#gid=175439716) if needed — schemas evolve.

## Notation conventions in the Sheet

A few things to know when reading the referential:

- **"Raw"** schemas (`RAW_INVENTORY`) hold data with minimal transformation from Amazon's APIs. Trade-off: more granular, but less curated. Most users should prefer the curated equivalents in `SELLING_PARTNER` or `FINANCE`.
- **"Preview"** schemas (e.g. `PREVIEW_RAW_SELLER`) are works in progress. Schema and content may change without notice; useful for early experiments, not for production queries.
- **Two `brand` columns** in some tables (`brand` and `brand_catalog`) reflect two distinct sources — Amazon public-data collection and Amazon API. Names sometimes diverge; both are exposed for completeness.
- **`_SRC` suffix** historically meant "source — minimally transformed". Some `_SRC` tables (e.g. `FBA_SNS_PERFORMANCE_SRC`) are now deprecated in favor of curated alternatives — see individual knowledge-hub pages for the migration paths.

## Frequently asked questions

<Accordions>

<Accordion title="How can I see sessions and page views for my products?">

This data lives in the `PREVIEW_RAW_SELLER` schema with sessions and page views broken down by platform (browser vs. mobile app), at <Term>SKU</Term>, child <Term>ASIN</Term>, and parent ASIN levels. For a more polished surface, the curated `SELLER_RETAIL_TRAFFIC_BY_ASIN_SRC` table in `SELLING_PARTNER` is the alternative.

</Accordion>

<Accordion title="How can I check which brands dominate a specific category?">

Use `REPORTS.REPORT_MARKET_BEST_SELLER_RANK_AND_ESTIMATES` — both `brand` and `brand_catalog` columns are populated for tracked categories.

</Accordion>

<Accordion title="How can I check which brands dominate search results for a specific keyword?">

Use `REPORTS.REPORT_SEO_PRODUCT_ORGANIC_RANK` (organic) and `REPORTS.REPORT_SEO_PRODUCT_SPONSORED_RANK` (sponsored, Premium Keywords only).

</Accordion>

<Accordion title="Why do some tables have two `brand` columns?">

Brand data comes from two sources — Amazon's website and Amazon's API. The two values sometimes diverge; both are exposed for completeness.

</Accordion>

<Accordion title="Why are some schemas labeled 'Raw'?">

"Raw" data is what we collect from Amazon's APIs with minimal transformation on DataHawk's side. Useful for granular analysis; not always the easiest surface to work with.

</Accordion>

<Accordion title="Why are some schemas labeled 'Preview'?">

"Preview" data is in active development — schema and content may change. Valuable for early experiments, but pin your production queries to non-preview tables when possible.

</Accordion>

<Accordion title="Can I get this data in a more raw / unaggregated format?">

Yes, in most cases. Reach out and we'll work with you on access to the raw layer.

</Accordion>

</Accordions>
